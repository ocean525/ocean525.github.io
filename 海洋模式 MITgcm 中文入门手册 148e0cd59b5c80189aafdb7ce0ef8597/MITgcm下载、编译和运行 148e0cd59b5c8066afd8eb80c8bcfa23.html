MITgcm下载、编译和运行

这个部分在官方英文手册的getting started部分，基本就是把重点步骤简单翻译过来了。

下载

有几种方式，随便用一种就行，下载的默认是最新版本，因为gcm是在发展的，所以也可以下载到一些以前的版本，详见官方手册。

  • 使用wget下载，直接在命令行输入：

    wget https://github.com/MITgcm/MITgcm/archive/master.zip

    下载完成之后解压即可，upzip master.zip

  • 使用git下载，有三种方式，
    • 在需要下载的目录下运行git clone https://github.com/MITgcm/MITgcm.git
    • 或者运行git clone git@github.com:MITgcm/MITgcm.git
    • 或者直接下载，简单粗暴

      进入https://github.com/MITgcm/MITgcm，如下图，点击绿色Code，在下方直接点击Download ZIP，就可以把模式下载到本地，再自行上传到服务器上解压即可。

下载并解压完成之后，可以看到以下目录：

各目录内容如下：

model：此目录包含主源代码，分为 src 和 inc 两个子目录。

eesupp：包含执行环境的源代码，同样分为 src 和 inc 两个子目录。

pkg：包含软件包源代码，每个包都有独立子目录。例如，pkg/seaice 是海冰模块代码，可以选择是否调用。

doc：包含 MITgcm 的文本格式文档。

tools：包含各种实用工具。其中 genmake2 是用 bash 编写的脚本，用于生成编译文件。tools/build_options 子目录包含"optfile"，提供了多种编译器和机器的编译选项。此目录还包含 tools/adjoint_options 和 tools/OAD_support 子目录，用于生成切线和伴随模型。

utils：包含多个实用程序。utils/matlab 子目录有用于直接读取模型输出到 MATLAB 的脚本，utils/python 包含 Python 相关程序，utils/shell 则包含用于合并处理器输出的 C-shell 后处理脚本。

verification：包含模型算例。

jobs：包含运行 MITgcm 的示例脚本。

lsopt：包含用于优化的行搜索代码。

optim：用于 MITgcm 和行搜索代码之间的接口。

源代码等部分平时用不上，一般是在verification目录下找到合适的算例，复制一份用于自由配置（不要直接修改原算例，否则你只能重新解压整个模式才能找到修改之前的算例），在自由配置的目录下就可以完成模式的编译、运行及读取等所有工作了。

编译

以exp2为例，进入exp2_test目录之后，可以看到一些目录，这些目录就构成了“一个具体的MITgcm模型”，如下图所示。

虽然有很多目录，但其实本质上只有两个目录是最基础的，那就是code和input，意思就是你只要有了这两个目录，那你就有了一个具体的模式配置，虽然它还没有被编译（在build目录下进行，生成可执行文件），也没有被运行（在run目录下进行，生成输出结果）,因此你会发现build目录和run目录目前都是空目录。

先不管三七二十一，接下来开始编译，进入build目录，你会发现这是一个空目录，首先需要确定环境配置好了，输入which icc，which ifort检查编译器是否在环境里面，配置好以后

  • 一、生成Makefile文件
~/MITgcm-master/tools/genmake2 -mods ../code/ -mpi -of ~/MITgcm-master/tools/build_options/linux_amd64_ifort+impi -rootdir ~/MITgcm-master/

除了../code以外，我给的都是绝对路径，这样可以保证在任何目录都能找到对应的文件，-rootdir是指定了gcm的根目录。运行结束之后会得到Makefile文件等。

  • 二、链接所需文件
make depend

运行结束之后，会将模型的源代码，code里面的配置文件等编译所需的所有文件链接过来，例如：

  • 三、编译
make

编译完成之后，会生成可执行文件mitgcmuv，至此，你的“一个具体的模型”就变成已编译未运行了。为什么要强调“具体的”呢？因为此时你生成的mitgcmuv，它是根据你的code目录下的内容生成的，code目录下包含了模型的网格信息，运行所需的核数等，这些都是很个性化的配置，你只能用指定的核数去运行这个mitgcmuv，并且也只能运行同样的网格，这也就意味着你的input里面的输入文件，必须要是这个网格，如果网格不匹配，是无法运行的。

简单来说，就是你生成的模式可执行文件是mitgcmuv，它的使用范围有限，只能在固定核数下运行，只能适配固定的网格。想修改核数或者网格，就必须从code目录下的SIZE.h开始修改，从头编译生成一个新的mitgcmuv文件。

运行

接下来可以开始运行了，进入run目录，运行

ln -s ../input/* .
cp ../build/mitgcmuv .
./mitgcmuv

这三行分别是，将input里面的所有文件链接过来、将build里面生成的mitgcmuv复制过来、运行mitgcmuv

值得注意的是，现在是用./mitgcmuv，也就是直接运行该文件，这就是单核运行，这是因为exp2的默认配置是单核，如果是多核，假如是4核，那就需要

mpirun -np 4 ./mitgcmuv

另外，这两种方式都是直接使用登录节点运行，一般而言登录节点是不能运行大型模式的，需要使用一些方式提交任务，例如sbatch等，但归根结底，是将这一行

mpirun -np 核数 ./mitgcmuv

提交到服务器上运行。

运行完成之后，可以看到NORMAL END，目录下就会生成结果文件。

  • 查看结果
vi STDOUT.0000
vi STDERR.0000

STDOUT是模式运行的过程，STDERR是模式运行中的错误。

T开头是温度文件，中间10个零是指它的输出步数，图里有0步、24步和26步，这是对应于input里面data中对输出时间的配置，

vi data

如下：

从0s开始运行，到2808000s结束，动量方程的时间步是2400s，示踪方程的时间步是108000s，输出时间间隔是2592000s，因此2592000/108000=24，在第24步输出了一次，而马上就到2808000s的结束时刻了，2808000/108000=26，因此在26步又输出了一次。我一般直接使用deltaT=***，时间参数的意义及给定见time-discretization、model-configuration等。

结果的读取，使用https://github.com/MITgcm/MITgcm/blob/master/utils/matlab/rdmds.m就可以读取结果文件，在模型目录下就有utils/matlab/rdmds.m，用法如下

% matlab
>> H=rdmds('Depth');
>> contourf(H');colorbar;
>> title('Depth of fluid as used by model');

>> eta=rdmds('Eta',24);
>> imagesc(eta');axis ij;colorbar;
>> title('Surface height at iter=10');

>> [eta,iters,M]=rdmds('Eta',NaN); % this will read all dumped iterations
>> % iter numbers put in variable 'iters'; 'M' is a character string w/metadata
>> for n=1:length(iters); imagesc(eta(:,:,n)');axis ij;colorbar;pause(.5);end
